## 2. 발견된 문제점

### 문제 1: 학원정보가 첫 화면

- 계약서는 **강사와의 계약**인데, 첫 화면이 **내 학원 정보 입력**
- 유료 회원(학원장)이 강사와 계약할 때마다 매번 학원정보를 입력해야 함
- DB에 `academy_info` 테이블이 있고 `user_id`별로 조회하지만, **자동 채움이 안됨** (수동 "불러오기" 버튼 클릭 필요)

### 문제 2: Step 3 (계약 유형 확인) 의미 없음

- 사용자 입력: **0개**
- 검증(validation): `canGoNext = true` (항상 통과)
- 스토어 연동: **없음** (읽기도, 쓰기도 안 함)
- 실질적으로 "다음" 버튼 1번 누르게만 하는 빈 화면

### 문제 3: 전체 확인 ↔ 미리보기 중복

- Step 8 (전체 확인): 입력 데이터를 요약 표시
- Preview (미리보기): 계약서 문서를 렌더링
- 두 화면 모두 같은 목적: **"제출 전 확인"** → 분리할 이유 없음

### 문제 4: 10단계는 너무 많음

- UX 연구 기준 멀티스텝 폼 최적 범위: **3~7단계** (Eleken, Creative Navy, NNGroup)
- 10단계는 사용자에게 심리적 부담 → 이탈률 증가

## 3. 개선안 A: 5단계 (채택)

### 새 흐름

```
 ① 강사정보 → ② 근무조건 → ③ 급여 → ④ 특약 → ⑤ 검토·완료
```

### 단계별 상세

#### ① 강사정보 (현재 Step 1)
- **입력**: 강사명*, 과목*, 주소, 연락처, 생년월일
- **변경**: 첫 화면으로 이동 (계약 대상부터 입력)
- **학원정보**: 설정/프로필에서 자동 로드 (계약서에 표시 안 됨 → 자동 반영)

#### ② 근무조건 (현재 Step 4 + Step 5 합체)
- **섹션 A - 계약기간**: 시작일, 기간, 종료일, 수습기간
- **섹션 B - 근무시간**: 요일별 출퇴근/휴게시간, 프리셋
- **이유**: 계약기간과 근무시간은 모두 "언제, 얼마나 일하는가"라는 같은 맥락

#### ③ 급여 (현재 Step 6 — 단독 유지)
- 급여유형 선택 (고정급/비율제/시급)
- 기본급, 수당 5종, 연장근로수당
- 경업금지 약정
- **이유**: 1000줄+ 복잡도, 자동 계산 다수 → 분리 필수

#### ④ 특약사항 (현재 Step 7 — 단독 유지)
- 특약사항 자유 입력 (선택)
- **이유**: 자유 텍스트 입력은 숫자 계산(급여)과 성격이 다름

#### ⑤ 검토·완료 (현재 Step 8 + Preview + Complete 합체)
- **탭 A - 요약**: 전체 데이터 요약 + 인라인 편집 (✏️ 클릭 시 해당 단계로 이동)
- **탭 B - 미리보기**: 계약서 문서 렌더링
- **액션**: 저장, 서명요청(카톡), PDF 다운로드, 링크 복사
- **이유**: "확인 → 미리보기 → 완료"를 하나의 화면에서 탭으로 전환

### 학원정보 처리 방식

```
[설정/프로필 페이지] (계약서 밖)
  └── 학원명, 주소, 대표자명, 연락처, 사업자번호, 직인
  └── Supabase academy_info 테이블에 저장
  └── 한 번 등록하면 계속 사용

[계약서 마법사 진입 시]
  └── academy_info 자동 조회 → wizard-store에 자동 세팅
  └── 사용자는 학원정보 입력 없이 바로 강사정보부터 시작
```

## 6. 기술 구현 계획

### Phase 1: 학원정보 분리
- `/settings` 또는 `/profile` 페이지 생성
- 기존 `page.tsx` (Step 0) 로직 → 설정 페이지로 이동
- 마법사 진입 시 `academy_info` 자동 조회 → store 세팅

### Phase 2: 스텝 재구성
- Step 3 (`step-3/`) 디렉토리 삭제
- Step 4 + Step 5 → 새 `step-2/` 합체 (섹션 A: 기간, 섹션 B: 시간)
- Step 6 → 새 `step-3/` (내용 동일)
- Step 7 → 새 `step-4/` (내용 동일)

### Phase 3: 검토·완료 합체
- Step 8 + Preview + Complete → 새 `step-5/` (탭 전환 UI)
- 요약 탭: 기존 Step 8 내용
- 미리보기 탭: 기존 Preview 내용
- 액션: 기존 Complete 로직 (저장, 서명요청, PDF)

### Phase 4: 인프라 업데이트
- `ProgressBar.tsx` — 5단계 그룹 반영
- `layout.tsx` — `TOTAL_STEPS = 5` 변경
- `wizard-store.ts` — 네비게이션 플로우 업데이트

## 7. 주의사항

### 데이터 의존성
- **② 근무조건** → **③ 급여**: 급여 계산에 주간근로시간(`weeklyHours`)이 필요
- 이 순서는 반드시 유지해야 함

### 기존 MVP 코드 위반 사항 (고도화 시 수정)
- `<select>`, `<input type="date">` 네이티브 폼 사용 중 (AGENTS.md 위반)
- `{showResumeModal && <Modal />}` 조건부 렌더링 패턴 (AGENTS.md 위반)
- `any` 타입 사용 (`preview/page.tsx`)

### Zustand Store
- `wizard-store.ts`의 persist 미들웨어 → localStorage 키 유지 필요
- 기존 사용자의 임시 저장 데이터 호환성 고려

## 8. 검토되었으나 채택하지 않은 안

### 안 B: 4단계 (불채택)
- 특약사항을 급여에 합치는 안
- **불채택 사유**: 급여 페이지가 이미 1000줄+로 복잡, 자유 텍스트와 숫자 계산은 성격이 다름

### 7단계안 (불채택)
- 학원정보만 프로필로 빼고 나머지 유지
- **불채택 사유**: Step 3 제거, 확인/미리보기 중복 해소만으로도 5단계 가능, 7단계도 여전히 많음
